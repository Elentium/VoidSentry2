VoidSentry2

Powerful & Performant buffer serialization library, successor to VoidSentry.

What it does

The library serializes Luau and Roblox data into buffer and reads it back. It supports 35+ types, an easy-to-use API, and optional Zstd compression via Roblox EncodingService.

Schematized Serialization

You define the way the data is serialized. The library writes and reads bytes in that fixed order — smaller payloads and predictable layout when both sides share the same schema.

Schemaless serialization

You pass plain values and the library infers their types at runtime, storing a small tag before each value. Handy when the list of values changes call to call without maintaining a separate schema for every combination.

Rich Types Support

Numbers, strings, vectors, CFrame, Color3, enum(...), arrays, maps, optional, any, instance (id map), and serInstance after registerInstanceSchema.

Configurable

Strict mode, dynamic instance/vector behavior, buffer cap — see the Configuration page for more info.

Quality Oriented

Strong Luau typing for clear autocomplete and safer refactors, hot paths tuned for performance, and a large built-in test suite run under stress to catch regressions.

Quick start

Static

--!strict
local VS = require(path.to.VoidSentry2)
local T = VS.types
local static = VS.static

local row = T.struct({
    n = T.int32,
    s = T.string,
})

local buf = static.serialize(row, { n = 42, s = "hi" }, nil, nil)
local out = static.deserialize(row, buf, nil, nil)

Dynamic

Each value you pass becomes one tagged chunk in the buffer. The receiver reads the same number of values in the same order.

local VS = require(path.to.VoidSentry2)
local dynamic = VS.dynamic

-- Example: send a name, a user id, a position, and whether sprint is on — no struct definition.
local playerId = 9001
local targetPos = Vector3.new(10, 0, -3)

local buf = dynamic.serialize(nil, nil, "MovePlayer", playerId, targetPos, true)
if not buf then return end

local name, id, pos, sprinting = dynamic.deserialize(nil, nil, buf)
-- name == "MovePlayer", id == 9001, pos == targetPos, sprinting == true

Configuration

Defaults live in the package; change them to match your game’s needs.

Where to edit

Main switches are in src/shared/config/constants.luau. If you use Wally, edit a vendored or forked copy of the library (or override via your own build pipeline) so updates do not wipe your changes. If you clone from GitHub, change the file in your repo before syncing to Studio.

Common options

  • strictMode — true: bad input usually throws. false: many failures warn and return nil.
  • maxBufferSize — Upper bound for the internal scratch buffer; serialization fails if a payload would exceed it.
  • dynamicInstanceType — In dynamic mode only: how a plain Instance is written. "instance" encodes a numeric id using _VSID and your instance map (same object on deserialize). "serInstance" uses registered schemas and property snapshots. Static schemas ignore this — they use whichever node you put in the struct (T.instance vs T.serInstance(...)).
  • dynamicVectorType — In dynamic mode only: how a vector object is inferred(luau vector or roblox Vector3)

Related code

src/shared/config/init.luau re-exports constants for the rest of the library. You normally do not require it from game code — only adjust constants.luau when you maintain your own copy of VoidSentry2.

API reference

Module Static Dynamic Instances Type catalog

Module exports

VoidSentry2.types

Table of type factories / nodes (int32, struct({ ... }), array(...), etc.). Each node exposes write / read internally; you pass nodes to static.serialize / deserialize.

VoidSentry2.static

staticSerializer with:

  • serialize<T>(serdes: typeNode<T>, input: T, offset: number?, compressionLevel: number?) -> buffer?
  • deserialize<T>(serdes: typeNode<T>, input: buffer, offset: number?, wasCompressed: boolean?) -> T?

VoidSentry2.dynamic

dynamicSerializer:

  • serialize<T...>(compressionLevel: number?, offset: number?, T...) -> buffer?
  • deserialize(wasCompressed: boolean?, offset: number?, input: buffer) -> ...any? — first argument wasCompressed: pass true if the buffer was produced with a non-nil compression level on serialize.

VoidSentry2.registerInstanceSchema(className: string, schema: { [string]: typeNode })

Registers a property layout for types.serInstance(className) and for dynamic inference when dynamicInstanceType is set to serInstance mode.

VoidSentry2.combine(t1: table, t2: table) -> table

Shallow merge of two tables (used for convenience; not serialization-specific).

Static serializer

Each call has exactly one root type node. To send several logical values in one go, put them in a single struct (or an array / map) — that is the normal pattern for “multiple fields” on the wire.

Offset

Writing starts at byte index offset (default 0) inside an internal scratch buffer. The returned buffer length equals the final write cursor. Bytes before offset are zero-filled in the returned buffer so you can overwrite them with a header, version, or magic value after serialize. On deserialize, pass the same start index so reading begins after your metadata.

Compression

If compressionLevel on serialize is a number, the whole payload is Zstd-compressed. On deserialize, pass wasCompressed = true to decompress first; pass nil or false for uncompressed buffers.

Dynamic serializer

Each value is stored with a small type id byte before its bytes so the reader knows what to decode next. If you used Zstd on serialize, pass wasCompressed = true on deserialize; otherwise pass nil or false.

Instances

types.instance

Serializes a registered Roblox Instance by numeric id (_VSID attribute + server/client instance map). Round-trip restores the same object reference.

types.serInstance("ClassName")

Requires a prior registerInstanceSchema. Produces a new instance on deserialize with properties read from the buffer.

Usage

How to think about the library before you copy examples — static vs dynamic, errors, compression, offsets, and config flags.

When VoidSentry2 helps

If you already send Luau tables through remotes or store big JSON-like structures, you pay for table encoding and larger network IO. A buffer is compact and fast to read back, as long as both sides agree on the layout (static schema) or on the order and compression flag (dynamic). It does not replace Roblox’s built-in replication; it is for your own byte payloads.

Type nodes and one root value

VoidSentry2.types exposes factories like T.uint32, T.struct({ ... }), T.array(T). Each value is a small object the library uses internally. You almost never call write/read yourself — you pass the node to static.serialize / static.deserialize. One serialize call = one root value (often one table shaped like your struct).

Static vs dynamic

Static — You pick the exact wire shape. Smallest size and least CPU usage when the schema is stable (player state, inventory snapshot, weapon stats). Use struct for “many fields in one message”.

Dynamic — You pass a variable list of values; each gets a one-byte type tag (plus payload). Easier for debug tools, admin commands, or “send whatever mix of primitives I have right now”. Payload is larger than static for the same data.

Strict vs lax (strictMode)

In src/shared/config/constants.luau, strictMode = true (default in source) makes most bad input throw. With false, many failures warn and serialize/deserialize return nil. Lax mode needs discipline: every call site should check for nil. Also remember optional(T) can legally deserialize to nil when the value is absent — that is not always an error.

Compression (Zstd)

Both serializers use Roblox EncodingService. Compression uses extra CPU on send and receive. On serialize you pass an optional numeric level; on deserialize you pass whether the buffer was compressed (see API reference). Try it on real payloads — small messages sometimes shrink little or not at all.

Offsets and custom headers

If you want some number of bytes of your own data at the start of the same buffer (for example a protocol version, magic value, or chunk length) before the VoidSentry2 payload, call serialize(..., headerByteCount, nil) then write your header into indices 0 .. headerByteCount - 1. Deserialize with the same offset so reading starts after your header. You can also keep headers in a separate buffer and pass only the payload with offset 0.

Examples

Player snapshot (static)

Assume local VS = require(...), local T = VS.types, local static = VS.static for each block below.

local T = VS.types
local static = VS.static

local Player = T.struct({
    id = T.uint32,
    name = T.string255,
    pos = T.vector3,
    hp = T.float32,
})

local buf = static.serialize(Player, {
    id = 1,
    name = "Aria",
    pos = Vector3.new(0, 10, 0),
    hp = 100,
}, nil, nil)

local round = static.deserialize(Player, buf, nil, nil)

Batched entities + Zstd

Serialize with a numeric level; deserialize with wasCompressed = true.

local EntityList = T.array(T.struct({
    id = T.uint16,
    x = T.float32,
    y = T.float32,
}))

local buf = static.serialize(EntityList, entities, nil, 5)
local out = static.deserialize(EntityList, buf, nil, true)

Offset: reserve bytes, then write a header

First HEADER bytes stay zero until you fill magic/version; payload starts at HEADER. Deserialize with the same offset.

local HEADER = 8
local Payload = T.struct({ score = T.uint16, name = T.string255 })

local buf = static.serialize(Payload, { score = 10, name = "Ann" }, HEADER, nil)
buffer.writeu32(buf, 0, 0x50505332) -- example magic "PPS2"
buffer.writeu32(buf, 4, 1)        -- example protocol version

local data = static.deserialize(Payload, buf, HEADER, nil)

Dynamic with Zstd

First argument to serialize is the compression level (or nil for none). First argument to deserialize is true if that buffer was compressed.

local dynamic = VS.dynamic

local bigString = string.rep("x", 4000)
-- First arg = Zstd level 4; then offset nil; then the values to pack.
local buf = dynamic.serialize(4, nil, "ServerLog", bigString)
if not buf then return end

-- Because we used a compression level above, pass true here.
local label, payload = dynamic.deserialize(true, nil, buf)
-- label == "ServerLog", payload == bigString

Static “RPC-shaped” packet (several fields, one struct)

local Rpc = T.struct({
    op = T.string255,
    playerId = T.uint32,
    amount = T.float32,
})

local buf = static.serialize(Rpc, {
    op = "Damage",
    playerId = 7,
    amount = 12.5,
}, nil, nil)

enum (Roblox EnumItem)

local MaterialField = T.enum(Enum.Material)
local Row = T.struct({ mat = MaterialField })

local buf = static.serialize(Row, { mat = Enum.Material.Slate }, nil, nil)
local back = static.deserialize(Row, buf, nil, nil)

Dynamic RPC-style tuple

Uncompressed dynamic buffer: first two arguments to serialize are nil (no Zstd, no write offset). Receiver unpacks in order.

local dynamic = VS.dynamic

local playerId = 42
local targetPos = Vector3.new(1, 0, 0)

local b = dynamic.serialize(nil, nil, "Move", playerId, targetPos, true)
if not b then return end

local opName, id, pos, wantsJump = dynamic.deserialize(nil, nil, b)
-- opName == "Move", id == playerId, pos == targetPos, wantsJump == true

Optional fields + nested struct

Model “maybe present” values with optional; nest struct for grouped fields.

local T = VS.types
local static = VS.static

local Profile = T.struct({
    userId = T.uint32,
    displayName = T.string255,
    bio = T.optional(T.string255),
    stats = T.struct({
        kills = T.uint16,
        deaths = T.uint16,
    }),
})

local buf = static.serialize(Profile, {
    userId = 9001,
    displayName = "Neo",
    bio = nil,
    stats = { kills = 12, deaths = 3 },
}, nil, nil)

String keys → numbers (map)

local T = VS.types
local static = VS.static

local Scores = T.map(T.string255, T.float32)

local buf = static.serialize(Scores, {
    ["arena_1"] = 99.5,
    ["arena_2"] = 12.0,
}, nil, nil)
local out = static.deserialize(Scores, buf, nil, nil)

void, nothing, and any

void is an empty table {} marker (zero payload bytes). nothing is Luau nil with zero bytes. any stores a discriminator plus an inner value — useful when one field can hold several shapes.

local Ping = T.struct({ kind = T.string255, payload = T.any })
local buf = static.serialize(Ping, { kind = "int", payload = 42 }, nil, nil)
local back = static.deserialize(Ping, buf, nil, nil)

serInstance: register schema, then serialize

Run registerInstanceSchema once per class name (usually at module load). Then use types.serInstance("ClassName") in your static schema. Deserialization creates a new Instance and assigns properties from the buffer.

local VS = require(game.ReplicatedStorage.VoidSentry2)
local T = VS.types
local static = VS.static

VS.registerInstanceSchema("Part", {
    Name = T.string255,
    Anchored = T.boolean,
})

local PartSnapshot = T.struct({
    template = T.serInstance("Part"),
})

local src = Instance.new("Part")
src.Name = "Floor"
src.Anchored = true

local buf = static.serialize(PartSnapshot, { template = src }, nil, nil)
local copy = static.deserialize(PartSnapshot, buf, nil, nil)
-- copy.template is a new Part with Name / Anchored restored

Mixing instance (by reference) and static structs

types.instance writes a numeric id and resolves the same object on the peer via your instanceMap and _VSID — use for live object graphs. serInstance snapshots property bags into new instances — use for templates or reconstructed props.

-- instance: same Part reference round-trips if both sides share instanceMap + _VSID
local RefPayload = T.struct({ target = T.instance })

-- serInstance: wire carries property values; deserialize builds a fresh Instance
local ClonePayload = T.struct({ cloneOfPart = T.serInstance("Part") })

Useful tips

Short habits that save time.

Build schemas once

Define local MySchema = T.struct({ ... }) at module scope. Creating new struct tables inside hot loops creates garbage and costs CPU.

Use smaller types when the data allows

If a number never exceeds 255, uint8 beats uint32. If position does not need full float precision, vector3f24 or vector3f16 saves bytes versus vector3.

Measure compression on real traffic

Zstd helps repetitive blobs; tiny messages may not shrink enough to pay for CPU.

Changing your schema breaks old bytes

Caution: adding/removing/reordering struct fields or switching node kinds changes the wire layout. Old clients or old save files will mis-read data if you do not migrate.

Tip: add a version = T.uint8 (or wider) field at the start of your root struct, bump it when you change layout, and branch reads or run a one-time migration.

nil from optional vs nil from failure

An absent optional(T) decodes as nil on success. In lax mode, failures can also yield nil. Use strictMode, outer framing (length + checksum), or protocol context to tell the cases apart when it matters.

Custom shapes & Roblox instances

Most of the time, combining built-in type nodes and calling registerInstanceSchema for serInstance is the right approach. Editing the library source is only for brand-new wire encodings.

1. Composable schemas (recommended): struct + primitives

What it is. You describe the wire layout by combining built-in nodes: struct, array, map, optional, numbers, strings, vectors, enum, any, etc. No changes to the VoidSentry2 package — only your game code.

When to use it. Almost always: gameplay state, RPC payloads, configs, save blobs with a stable field set. If the shape varies at runtime, combine optional, any, or the dynamic serializer instead of inventing a new primitive.

local T = VS.types
local static = VS.static

-- "Custom" in the sense of your game: a fixed record + list of sub-records
local InventorySlot = T.struct({
    itemId = T.uint16,
    count = T.uint8,
})

local PlayerSave = T.struct({
    version = T.uint8,
    coins = T.uint32,
    slots = T.array(InventorySlot),
})

local buf = static.serialize(PlayerSave, saveTable, nil, nil)

2. Extending the library (fork): buffer primitives vs type nodes

VoidSentry2 splits raw bytes and named wire types into two layers. They are related but not the same file.

2a. Primitive encodings — src/buffer/writer.luau & src/buffer/reader.luau

What they are. Low-level helpers (writef32, writef24, writeu16, NEED, START/FINISH, etc.) that advance the shared scratch buffer cursor and read/write bytes. They do not assign type discriminators and are not ”VoidSentry2 types” — they are building blocks.

When to change them. When you need a new byte layout that does not exist yet (e.g. another packed float size, a custom fixed-width blob). Pair every new write* with a matching read* in the reader.

Example in-tree. writef24 / the matching read path implement the float24 wire format; src/types/float24.luau then calls those primitives from its type node’s write/read.

2b. Type nodes — src/types/<name>.luau + src/shared/config/typeSigns.luau

What they are. Each file exports a frozen typeNode: { write = …, read = … } that validates Luau/Roblox values, calls writer.NEED, then buffer primitives. This is where float32, struct, qcframe, etc. live.

How they are registered (important). Root src/init.luau loops every entry in typeSigns.luau and does require("@self/types/" .. name). The key in typeSigns must match the filename (e.g. float32 = 11 → src/types/float32.luau). Those modules fill signMap and the exported types table. The small src/types/init.luau file only declares empty tables; it does not register types — that happens in src/init.luau.

What you add for a new wire type. (1) A new unique byte id in typeSigns.luau. (2) A new src/types/<name>.luau whose name matches that key, implementing write/read like src/types/uint8.luau or src/types/float32.luau.

2c. Dynamic serializer & inference

In dynamic mode, dynanode writes a one-byte type sign (from typeSigns) before each value’s payload. If your new type should be chosen automatically from plain Luau values, you may also need to extend src/shared/inferValueType.luau. Static schemas call the node directly and do not emit that per-root sign — they still use the same typeNode tables.

When to bother. Only for wire formats or runtime inference you cannot build from existing nodes (section 1). Otherwise prefer composition and keep upstream merges easy.

-- Conceptual split:

-- src/buffer/writer.luau  →  add writeMyBits(...) / reader symmetry
-- src/types/myformat.luau →  write = function(v) NEED(n) … writeMyBits(v) … end
-- src/shared/config/typeSigns.luau →  myformat = 40  -- next free id

-- Optional for dynamic auto-pick:
-- src/shared/inferValueType.luau → map typeof(value) / tags to typeSigns.myformat

3. serInstance end-to-end

types.serInstance(className) serializes a fixed set of properties you declare per class. Flow:

  1. Register once per class name. VoidSentry2.registerInstanceSchema(className, { PropName = typeNode, ... }) — keys are property names as strings; values are type nodes (e.g. T.string255, T.vector3). Call during server/client startup before any serialize.
  2. Reference in schemas. Use T.serInstance("Part") (same string as className) inside
  3. Serialize. Pass a real Instance of that class; the library reads each registered property from the object and writes bytes.
  4. Deserialize. The library creates a new Instance with Instance.new(className), then assigns properties from the buffer. It is not the same object identity as the source.

Contrast with types.instance. That encodes a reference (numeric id + _VSID + shared instanceMap) so the same Instance round-trips on both sides. Use references for live replication; use serInstance when you want a property snapshot or a fresh clone from wire data.
